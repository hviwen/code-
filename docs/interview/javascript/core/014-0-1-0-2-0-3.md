### 知识点系统梳理

**精度问题的根本原因：**
- JavaScript使用IEEE 754双精度浮点数
- 二进制无法精确表示某些十进制小数
- 0.1在二进制中是无限循环小数

### 实战应用举例
```javascript
// 1. 精度问题演示和解决
console.log(0.1 + 0.2);                    // 0.30000000000000004
console.log(0.1 + 0.2 === 0.3);            // false

// 解决方案：使用Number.EPSILON
function floatEqual(a, b) {
  return Math.abs(a - b) < Number.EPSILON;
}

console.log(floatEqual(0.1 + 0.2, 0.3));   // true
```

```javascript
// 2. 金融计算的安全方法
function safeAdd(a, b) {
  // 转换为整数计算
  const factor = Math.pow(10, Math.max(
    (a.toString().split('.')[1] || '').length,
    (b.toString().split('.')[1] || '').length
  ));

  return (Math.round(a * factor) + Math.round(b * factor)) / factor;
}

console.log(safeAdd(0.1, 0.2));            // 0.3
console.log(safeAdd(0.1, 0.2) === 0.3);    // true
```

### 记忆要点总结
- **问题原因**：IEEE 754标准，二进制无法精确表示十进制小数
- **典型例子**：0.1 + 0.2 !== 0.3
- **解决方案**：Number.EPSILON比较、整数计算、第三方库
- **实际应用**：金融计算中需要特别注意精度问题

## 使用场景说明和对比

| 场景 | 推荐方案 | 原因 |
| --- | --- | --- |
| 展示普通小数 | `toFixed` / `Intl.NumberFormat` | 控制展示精度 |
| 判断近似相等 | 误差范围比较 | 避免直接 `===` |
| 金额计算 | 转成分等整数单位或用 decimal 库 | 避免浮点误差影响账务 |
| 科学计算 | 相对误差比较 | 大小量级不同，固定 EPSILON 不够 |

前端展示可以格式化，业务计算要看风险。金额、库存、积分这类影响资产的数据，不应只靠浮点数直接加减。

## 易错点提示

1. 不是 JavaScript 独有问题，使用 IEEE 754 二进制浮点的语言都有类似现象。
2. `0.1 + 0.2` 的结果是 `0.30000000000000004`，所以严格等于 `0.3` 为 false。
3. `toFixed` 主要解决展示，不等于从根上解决计算精度。
4. `Number.EPSILON` 适合小数近似比较，不适合所有量级。
5. 金额计算优先使用整数单位或专门 decimal 库。

## 记忆要点总结

- JS Number 是 IEEE 754 双精度浮点数。
- 某些十进制小数无法被二进制精确表示。
- 小数比较不要直接 `===`。
- 金额类计算要用整数单位或高精度方案。

## 可能类似的问题及简要参考答案

**Q：`0.1 + 0.2` 为什么不是精确的 0.3？**  
A：0.1 和 0.2 转成二进制都是无限循环小数，按有限位存储后会产生舍入误差。

**Q：如何让页面显示 0.30？**  
A：展示层可以用 `(0.1 + 0.2).toFixed(2)` 或 `Intl.NumberFormat`，但这只是格式化显示。

## 辅助记忆总结

一句话记：十进制小数进二进制会有舍入，展示可格式化，计算要控误差。
