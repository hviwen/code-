# 020. [中级] 闭包的实际应用场景有哪些？

> 来源：`docs/javascript/js_interview_questions_part_1.md`

## 问题本质解读

闭包的价值不是“函数里套函数”，而是内部函数能持续访问外层函数已经创建的词法环境。它常用来保存状态、隐藏变量、延迟执行逻辑和生成定制函数。

一句话答法：闭包让函数带着创建时的上下文继续运行，所以适合做私有状态、函数工厂、防抖节流、缓存和事件回调。

## 问题意图

这道题想确认你是否能把闭包从概念迁移到项目代码里：知道闭包能解决什么问题，也知道它会让被引用的数据延长生命周期。

## 考察范围

- 词法作用域和作用域链。
- 闭包形成条件：函数创建 + 引用外层变量 + 外部继续持有该函数。
- 私有变量、函数工厂、防抖节流、缓存、事件处理。
- 闭包与内存释放、垃圾回收的关系。
- 闭包和 class 私有字段、模块作用域的取舍。

## 技术错误纠正

原始内容列出了“防抖、节流、高阶函数”，方向是对的，但不完整。更准确的表达是：防抖节流只是闭包保存状态的一个应用，闭包本身是词法环境被函数引用后继续存在的机制。

## 知识点系统梳理

闭包通常由三部分组成：

1. 外层函数创建局部变量。
2. 内层函数引用这些变量。
3. 内层函数被返回、传递或注册到外部，导致变量无法在外层函数结束后立刻释放。

常见应用：

| 应用 | 闭包保存什么 | 价值 |
| --- | --- | --- |
| 私有变量 | 内部状态 | 外部只能通过暴露的方法访问 |
| 函数工厂 | 配置参数 | 生成定制函数，减少重复参数 |
| 防抖节流 | 定时器 id、上次执行时间 | 控制高频事件触发 |
| 缓存/记忆化 | 缓存表 | 避免重复计算 |
| 事件回调 | 注册时的上下文 | 事件发生时仍能访问当时数据 |

闭包不是免费的：它会延长被引用变量的生命周期。如果闭包长期挂在 DOM 事件、定时器或全局缓存上，被引用的大对象也会一直留在内存中。

## 实战应用举例

### 示例 1：搜索框防抖

这个例子证明闭包可以保存 `timer`，让多次输入共享同一个定时器状态。

```js
function debounce(fn, delay) {
  let timer = null

  return function debounced(...args) {
    clearTimeout(timer)
    timer = setTimeout(() => {
      fn.apply(this, args)
    }, delay)
  }
}

const search = debounce(keyword => {
  console.log('request:', keyword)
}, 300)

search('j')
search('js')
search('javascript')
```

边界：闭包保存的是同一个 `timer`。如果组件卸载时还有事件监听器引用 `search`，要移除监听器，否则闭包仍可能存在。

### 示例 2：函数工厂保存配置

```js
function createPermissionChecker(requiredRole) {
  return function canAccess(user) {
    return user.roles.includes(requiredRole)
  }
}

const canEdit = createPermissionChecker('editor')

console.log(canEdit({ roles: ['viewer', 'editor'] })) // true
console.log(canEdit({ roles: ['viewer'] })) // false
```

这里闭包保存 `requiredRole`，后续只传用户对象即可，适合权限判断、表单校验、接口参数预设等场景。

## 使用场景说明和对比

| 场景 | 适合闭包吗 | 原因 |
| --- | --- | --- |
| 防抖、节流 | 适合 | 需要跨调用保存定时器或时间戳 |
| 创建私有计数器 | 适合 | 外部不能直接修改内部状态 |
| 大量长期缓存 | 谨慎 | 可能造成内存占用，需要淘汰策略 |
| 复杂业务对象建模 | 不一定 | class、模块或状态管理可能更清晰 |
| 一次性计算 | 不需要 | 普通局部变量即可 |

和相近方案对比：

| 方案 | 优点 | 代价 |
| --- | --- | --- |
| 闭包 | 简单、轻量、天然私有 | 状态隐藏后不易调试，可能延长内存生命周期 |
| class 私有字段 | 结构清晰，适合对象模型 | 写法更重 |
| 模块作用域 | 适合共享单例状态 | 多实例隔离不如闭包自然 |

## 易错点提示

- 不是所有内部函数都是闭包；关键是内部函数引用外部变量并被外部持有。
- 闭包保存的是变量引用，不是创建时的值快照。
- 循环里使用 `var` 创建回调会共享同一个变量，`let` 会为每次迭代创建新的绑定。
- 闭包能隐藏状态，也会让状态更难观察和释放。
- 防抖节流中的清理函数很重要，特别是组件卸载、路由切换时。

## 记忆要点总结

- 闭包 = 函数 + 创建时可访问的词法环境。
- 实际价值：保存状态、隐藏变量、延迟执行、生成定制函数。
- 高频场景：防抖节流、缓存、事件回调、函数工厂。
- 风险：被闭包引用的数据生命周期会变长。
- 回答时一定补一个“保存了什么状态”的例子。

## 可能类似的问题及简要参考答案

**Q：闭包是什么？**  
A：函数引用了外层词法作用域中的变量，并且函数在外层函数结束后仍被持有，这个函数和它引用的环境就形成闭包。

**Q：闭包能实现私有变量吗？**  
A：可以。把变量放在外层函数局部作用域，只返回访问或修改它的方法，外部无法直接访问变量本身。

**Q：闭包保存的是值还是引用？**  
A：保存的是变量绑定。变量后续变化时，闭包读取到的是同一个绑定的当前值。
