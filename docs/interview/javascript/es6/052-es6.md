# 052. [中级] 如何理解变量提升在 ES6 中的变化？

> 来源：`docs/javascript/js_interview_questions_part_2.md`

## 问题本质解读

ES6 并不是取消了变量提升，而是通过 `let`、`const` 和暂时性死区让“声明前使用变量”从静默得到 `undefined` 变成尽早报错。

一句话回答：`var` 会提升并初始化为 `undefined`；`let`、`const` 也会提升，但在执行到声明语句前不会初始化，访问会触发 TDZ 的 `ReferenceError`。

## 问题意图

面试官想看你能否纠正一个常见误解：“`let`/`const` 不存在变量提升”。如果你能解释 TDZ，就说明你理解编译阶段创建绑定和运行阶段初始化之间的区别。

## 考察范围

- `var` 的声明提升和初始化。
- `let`、`const` 的声明提升与 TDZ。
- 函数声明的提升。
- 重复声明、块级作用域带来的安全性变化。

## 技术错误纠正

“`let` 和 `const` 防止变量提升到全局”这个说法不准确。它们不是防止提升，而是把变量限制在块级作用域，并且在初始化前禁止访问。

“ES6 之后变量不再提升”也不准确。否则下面代码中的 `typeof value` 应该返回 `'undefined'`，但实际会因为 TDZ 抛错。

## 知识点系统梳理

| 声明方式 | 是否创建绑定 | 初始化时机 | 声明前访问 |
| --- | --- | --- | --- |
| `var` | 是 | 进入作用域时初始化为 `undefined` | 得到 `undefined` |
| `let` | 是 | 执行到声明语句时初始化 | `ReferenceError` |
| `const` | 是 | 执行到声明语句并完成赋值时初始化 | `ReferenceError` |
| 函数声明 | 是 | 进入作用域时完成初始化 | 通常可提前调用 |

TDZ 的重点不是“变量不存在”，而是“绑定已经存在，但还不能访问”。

## 实战应用举例

```js
console.log(a) // undefined
var a = 1
```

这段代码可以理解为：

```js
var a
console.log(a)
a = 1
```

但 `let` 不同：

```js
console.log(b) // ReferenceError
let b = 1
```

`b` 的绑定已经在词法环境中创建，所以它不会去外层找同名变量；但声明语句执行前，`b` 处于 TDZ，访问就会报错。

```js
const count = 1

if (true) {
  // console.log(count) // ReferenceError
  const count = 2
  console.log(count) // 2
}
```

块内的 `const count` 会遮蔽外层 `count`。从块开始到声明执行前都属于内层 `count` 的 TDZ，因此不会读取外层的 `count`。

## 使用场景说明和对比

ES6 的变化让错误更早暴露。以前 `var` 声明前访问会得到 `undefined`，后续可能在业务逻辑里变成隐蔽 bug；现在 `let`、`const` 会直接报错，开发阶段更容易定位。

```js
function getDiscount(price) {
  if (price > threshold) {
    return 0.8
  }

  const threshold = 100
  return 1
}
```

这个函数会在访问 `threshold` 时抛错。虽然外层可能也有 `threshold`，但块内声明会形成遮蔽和 TDZ。面试回答里要能说清楚：这是词法作用域规则，不是运行时“找不到变量”那么简单。

| 场景 | `var` 行为 | `let`/`const` 行为 | 复习重点 |
| --- | --- | --- | --- |
| 声明前读取 | 返回 `undefined` | TDZ 报错 | 提升不等于可用 |
| 块内同名变量 | 可能泄漏函数作用域 | 遮蔽外层并形成 TDZ | 词法作用域优先 |
| 重复声明 | 同一作用域可重复 | 同一作用域报错 | 避免意外覆盖 |
| 调试未初始化变量 | 错误可能延后 | 更早暴露 | 安全性提升 |

## 易错点提示

不要说 `let`、`const` 不提升。更准确的表达是：提升但不初始化，TDZ 阶段不能访问。

不要把 `typeof` 当作永远安全的检测方式。对未声明变量使用 `typeof` 是 `'undefined'`，但对 TDZ 中的变量使用 `typeof` 会报错。

不要忽略函数声明。函数声明通常会整体提升，但块级函数声明在严格模式和非严格模式下还有额外差异。

## 记忆要点总结

- `var`：提升并给 `undefined`。
- `let`/`const`：提升但先锁住。
- TDZ：绑定存在，但声明执行前不能碰。
- ES6 的目的：减少声明前误用，提前暴露错误。

## 延伸问题

为什么 TDZ 中的变量不会去外层作用域查找同名变量？

参考答案：词法作用域在代码结构上已经确定。当前块中存在同名声明时，该名字会绑定到当前块的词法环境；即使还处于 TDZ，也不会跳过当前绑定去读取外层变量。

## 可能类似的问题及简要参考答案

问：`typeof undeclared` 和 `typeof tdzVariable` 有什么区别？

答：前者变量没有声明，返回 `'undefined'`；后者变量已在当前词法环境中声明但未初始化，处于 TDZ，会抛 `ReferenceError`。

问：`const` 为什么必须声明时赋值？

答：`const` 绑定初始化后不能重新赋值，如果声明时不赋值，就没有合法机会完成初始化。

## 辅助记忆总结

用一句话记：ES6 不是“不提升”，而是“先登记名字，等声明执行前不准用”。
