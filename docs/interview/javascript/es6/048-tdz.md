# 048. [中级] 什么是暂时性死区（TDZ）？

> 来源：`docs/javascript/js_interview_questions_part_2.md`

## 问题本质解读

TDZ 是 `let`、`const` 声明从进入作用域到真正执行声明语句之前的区域。在这个区域访问变量会抛 `ReferenceError`。

一句话答法：变量已经被作用域记录了，但还没初始化，所以不能访问。

## 问题意图

这道题考察你是否理解 `let/const` 不是“没有提升”，而是提升后处于不可访问状态。

## 考察范围

- `let/const` 的声明提升。
- 初始化前访问报错。
- TDZ 与块级作用域。
- `typeof` 在 TDZ 中也会报错。
- 默认参数中的 TDZ。
- TDZ 对代码安全性的意义。

## 技术错误纠正

不要说 `let/const` 不会提升。更准确是：它们会创建词法绑定，但在声明执行前处于 TDZ，不能读取或写入。

## 知识点系统梳理

```js
{
  console.log(name) // ReferenceError
  let name = 'Ada'
}
```

执行进入块时，`name` 绑定已存在，但未初始化。直到执行到 `let name = 'Ada'` 后，TDZ 才结束。

`typeof` 也不安全：

```js
{
  console.log(typeof value) // ReferenceError
  let value = 1
}
```

## 实战应用举例

### 示例 1：TDZ 暴露错误访问顺序

```js
function createUser() {
  const user = {
    name,
  }

  const name = 'Ada'
  return user
}

createUser() // ReferenceError
```

这个例子证明：TDZ 可以提前暴露“先使用后声明”的错误。

### 示例 2：默认参数中的 TDZ

```js
function demo(a = b, b = 1) {
  return a + b
}

demo() // ReferenceError
```

参数从左到右初始化，`a = b` 时 `b` 还在 TDZ。

## 使用场景说明和对比

| 写法 | 访问声明前变量 | 结果 |
| --- | --- | --- |
| `var a` | 允许 | `undefined` |
| `let a` | 不允许 | `ReferenceError` |
| `const a` | 不允许 | `ReferenceError` |
| 未声明变量 | 不允许 | `ReferenceError` |

TDZ 不是为了增加麻烦，而是为了避免 `var` 那种静默得到 `undefined` 的隐藏 bug。

## 易错点提示

- `let/const` 也会提升，只是不能在初始化前访问。
- `typeof` 遇到 TDZ 变量也会报错。
- TDZ 起点是进入作用域，不是代码执行到声明上一行。
- 默认参数也有自己的作用域和 TDZ。
- `const` 声明必须初始化，否则语法错误。

## 记忆要点总结

- TDZ = 绑定已存在但未初始化。
- 访问 TDZ 变量会 `ReferenceError`。
- `typeof` 不再总是安全。
- TDZ 和块级作用域一起工作。
- 它让“先声明再使用”变成强约束。

## 可能类似的问题及简要参考答案

**Q：TDZ 从哪里开始？**  
A：从进入当前词法作用域开始，到执行声明语句完成初始化结束。

**Q：TDZ 中访问变量报什么错？**  
A：`ReferenceError`。

**Q：`var` 有 TDZ 吗？**  
A：没有。`var` 提升后会初始化为 `undefined`。

## 辅助记忆总结

记成一句话：TDZ 是“名字登记了，但还没激活，不能碰”。
