# 053. [高级] 解释 ES6 中函数声明在块级作用域中的行为

> 来源：`docs/javascript/js_interview_questions_part_2.md`

## 问题本质解读

这道题考察块级函数声明。它之所以高级，是因为函数声明在块内既受 ES6 块级作用域影响，又受浏览器兼容历史影响，严格模式和非严格模式的行为不同。

一句话回答：严格模式下，块级函数声明只在块内可见，并在块内提升；非严格模式下浏览器为了兼容旧代码，可能还会在外层创建同名绑定，行为更复杂，项目中应优先用函数表达式。

## 问题意图

面试官想判断你是否知道“函数声明会提升”并不是一句话能覆盖所有情况。块级函数声明涉及模式差异、兼容语义和可维护性取舍。

## 考察范围

- 函数声明提升。
- 块级作用域。
- 严格模式与非严格模式差异。
- 为什么不建议在 `if`、`for` 等块中写函数声明。

## 技术错误纠正

“块内函数外部一定不能访问”只适用于严格模式下的标准理解。非严格模式下，为了兼容历史浏览器行为，块级函数声明可能会在外层作用域形成类似 `var` 的绑定。

“块级函数声明和 `const fn = () => {}` 完全一样”也不准确。函数表达式遵循变量声明规则，更可预测；块级函数声明有特殊兼容规则。

## 知识点系统梳理

| 场景 | 块内是否可用 | 块外是否可用 | 推荐程度 |
| --- | --- | --- | --- |
| 严格模式块级函数声明 | 可用，块内提升 | 不可用 | 理解即可，少写 |
| 非严格模式块级函数声明 | 可用 | 可能可用，行为复杂 | 避免 |
| `const fn = function () {}` | 声明后可用 | 取决于声明位置 | 推荐 |
| `const fn = () => {}` | 声明后可用 | 取决于声明位置 | 推荐，但注意 `this` |

块级函数声明可以理解成“块内有函数声明提升”，但不要把它简单等同于顶层函数声明。

## 实战应用举例

严格模式下：

```js
'use strict'

if (true) {
  console.log(run()) // ok，块内函数声明提升

  function run() {
    return 'block'
  }
}

// console.log(run()) // ReferenceError
```

函数 `run` 在 `if` 块内可用，块外不可见。

非严格模式下不要依赖块外访问：

```js
if (true) {
  function run() {
    return 'block'
  }
}

console.log(typeof run) // 不同环境和历史语义下容易产生理解成本
```

为了避免模式差异，业务代码更推荐这样写：

```js
const createRunner = (enabled) => {
  if (!enabled) {
    return null
  }

  return function run() {
    return 'block'
  }
}

const runner = createRunner(true)
if (runner) {
  console.log(runner())
}
```

这里函数是否存在由返回值表达，不依赖块级函数声明的特殊提升。

## 使用场景说明和对比

块级函数声明适合理解语言机制，不适合作为团队代码风格。因为读者需要同时判断当前文件是否严格模式、构建产物是否转译、运行环境是否有历史兼容行为。

如果是“条件性定义函数”，更清晰的写法是：

```js
const normalize = needTrim
  ? (value) => value.trim()
  : (value) => value
```

如果函数逻辑较复杂，可以先声明稳定函数，再在条件中选择引用：

```js
function trimValue(value) {
  return value.trim()
}

function keepValue(value) {
  return value
}

const normalize = needTrim ? trimValue : keepValue
```

这种写法没有块级函数声明的兼容问题，测试和阅读都更直接。

| 写法 | 可读性 | 主要风险 | 推荐结论 |
| --- | --- | --- | --- |
| 块级函数声明 | 依赖模式判断 | 严格/非严格模式差异 | 面试会答，业务少写 |
| 函数表达式赋给 `const` | 初始化时机明确 | 声明前不可访问 | 推荐 |
| 条件返回函数 | 函数存在性由返回值表达 | 需要处理 `null` | 推荐 |
| 提前声明多个函数再选择 | 最稳定 | 代码略长 | 复杂逻辑推荐 |

## 易错点提示

不要在回答里只说“函数声明会提升”，要补一句“块级函数声明在严格模式下只提升到块内”。

不要依赖非严格模式下块外访问函数的行为。即使某些环境可行，也会让维护者难以判断。

不要把箭头函数作为所有替代答案。箭头函数没有自己的 `this`、`arguments`，如果函数体依赖这些语义，应使用普通函数表达式。

## 记忆要点总结

- 严格模式：块内函数，块内提升，块外不可见。
- 非严格模式：兼容历史，块外行为复杂。
- 项目实践：少写块级函数声明，多用函数表达式。
- 回答重点：模式差异 + 不建议依赖。

## 延伸问题

为什么在团队规范里常常禁止块级函数声明？

参考答案：因为它同时受严格模式、非严格模式、转译结果和运行环境影响，读者需要额外推理。用函数表达式或提前声明函数可以让作用域和初始化时机更明确。

## 可能类似的问题及简要参考答案

问：块级函数声明和函数表达式有什么区别？

答：块级函数声明有特殊提升和兼容规则；函数表达式遵循 `let`/`const` 的声明规则，初始化前不能访问，行为更可预测。

问：`if` 里面能不能写函数声明？

答：语法上可以，但不建议。严格模式下只在块内可见，非严格模式下兼容行为复杂，业务代码更推荐函数表达式或提前声明。

## 辅助记忆总结

用一句话记：块级函数声明“严格模式像块内函数，非严格模式像历史包袱”，会答即可，少在业务里写。
