# 046. [初级] `let`、`const` 和 `var` 的区别

> 来源：`docs/javascript/js_interview_questions_part_2.md`

## 问题本质解读

`let`、`const`、`var` 的核心区别是作用域、提升行为、重复声明和可重新赋值规则。现代 JavaScript 默认优先 `const`，需要重新赋值才用 `let`，很少再主动使用 `var`。

一句话答法：`var` 是函数作用域且会提升为 `undefined`；`let/const` 是块级作用域，有暂时性死区；`const` 不能重新绑定。

## 问题意图

这道题考察 ES6 变量声明的基础质量：是否理解块级作用域、TDZ、重复声明风险，以及项目里为什么推荐 `const` 优先。

## 考察范围

- 函数作用域和块级作用域。
- 变量提升和暂时性死区。
- 重复声明。
- 重新赋值和引用类型修改。
- 全局声明对 `window` 的影响。
- 现代编码规范中的选择顺序。

## 技术错误纠正

不能简单说 `const` 声明的值“不能变”。准确说法是：`const` 绑定不能重新赋值；如果绑定的是对象，对象内部属性仍可能被修改。

## 知识点系统梳理

| 对比项 | `var` | `let` | `const` |
| --- | --- | --- | --- |
| 作用域 | 函数作用域 | 块级作用域 | 块级作用域 |
| 变量提升 | 提升并初始化为 `undefined` | 提升但处于 TDZ | 提升但处于 TDZ |
| 重复声明 | 同作用域允许 | 同作用域不允许 | 同作用域不允许 |
| 重新赋值 | 允许 | 允许 | 不允许重新绑定 |
| 声明时初始化 | 可不初始化 | 可不初始化 | 必须初始化 |
| 全局声明 | 可能挂到 `window` | 不挂到 `window` | 不挂到 `window` |

## 实战应用举例

### 示例 1：块级作用域

```js
if (true) {
  var a = 1
  let b = 2
  const c = 3
}

console.log(a) // 1
console.log(b) // ReferenceError
console.log(c) // ReferenceError
```

这个例子证明：`let/const` 被限制在块级作用域，能减少变量泄漏。

### 示例 2：const 不能重新绑定，但对象属性可改

```js
const user = { name: 'Ada' }

user.name = 'Grace'
console.log(user.name) // Grace

user = { name: 'Lin' } // TypeError
```

## 使用场景说明和对比

| 场景 | 推荐 | 原因 |
| --- | --- | --- |
| 值不需要重新赋值 | `const` | 表达“绑定不变” |
| 循环计数器、后续要改值 | `let` | 块级作用域且允许改 |
| 老代码兼容 | `var` | 只在维护旧代码时遇到 |
| 常量对象 | `const` + 必要时 `Object.freeze` | `const` 不等于深不可变 |

## 易错点提示

- `let/const` 也会提升，只是初始化前处于 TDZ。
- `const` 限制的是绑定，不是对象内容。
- `var` 没有块级作用域，循环和条件里容易泄漏。
- 同一作用域里 `let/const` 不能重复声明。
- 全局 `var` 可能成为 `window` 属性，`let/const` 不会。

## 记忆要点总结

- 默认 `const`，需要重新赋值才 `let`。
- 避免主动写 `var`。
- `let/const` 是块级作用域。
- `const` 不等于对象不可变。
- TDZ 能提前暴露错误用法。

## 可能类似的问题及简要参考答案

**Q：`let` 和 `const` 会变量提升吗？**  
A：会创建绑定，但初始化前处于 TDZ，访问会报错。

**Q：什么时候用 `let`？**  
A：变量确实需要重新赋值时，比如循环计数器或状态累积。

**Q：`var` 为什么不推荐？**  
A：函数作用域、可重复声明、提升为 `undefined`，容易隐藏错误。

## 辅助记忆总结

记成一句话：`const` 管绑定，`let` 管变化，`var` 管旧代码。
