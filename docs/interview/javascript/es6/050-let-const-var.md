# 050. [初级] 在什么情况下使用 `let`、`const`、`var`？

> 来源：`docs/javascript/js_interview_questions_part_2.md`

## 问题本质解读

这道题不是让你背“优先 const”，而是考察你能否用变量声明表达代码意图：这个绑定会不会重新赋值、作用域应不应该被限制、是否需要兼容旧环境。

一句话回答：现代 JavaScript 中默认用 `const`，确实需要重新赋值时用 `let`，通常避免 `var`，除非维护旧代码或刻意依赖函数作用域。

## 问题意图

面试官通常想确认三件事：

1. 你是否理解 `const` 限制的是“绑定”，不是对象内容。
2. 你是否能用 `let` 表达变量后续会变化，而不是随手声明。
3. 你是否知道 `var` 的函数作用域和变量提升会带来可读性风险。

## 考察范围

- `var`、`let`、`const` 的作用域差异。
- 是否允许重复声明、是否存在暂时性死区。
- `const` 对引用类型的限制边界。
- 项目规范中的 `prefer-const`、`no-var`。

## 技术错误纠正

`var` 不是“声明全局变量”的同义词。它是函数作用域：在函数内声明时只属于函数，在全局脚本中用 `var` 声明才可能挂到全局对象上。

`const` 也不是“值不可变”。它只保证变量名不能重新绑定到另一个值；对象、数组内部成员仍然可以被修改。

## 知识点系统梳理

| 声明方式 | 作用域 | 可重新赋值 | 可重复声明 | 声明前访问 | 推荐程度 |
| --- | --- | --- | --- | --- | --- |
| `const` | 块级作用域 | 否 | 否 | TDZ 报错 | 默认首选 |
| `let` | 块级作用域 | 是 | 否 | TDZ 报错 | 需要变化时使用 |
| `var` | 函数作用域 | 是 | 是 | `undefined` | 旧代码中才保留 |

实战选择可以按这个顺序判断：

1. 变量后续不会重新赋值：用 `const`。
2. 变量后续确实会重新赋值：用 `let`。
3. 需要兼容极旧环境或维护老代码：才考虑 `var`。

## 实战应用举例

```js
function buildUserCard(user, permissions) {
  const displayName = user.nickname || user.name
  const canEdit = permissions.includes('edit:user')

  let statusText = '离线'
  if (user.online) {
    statusText = '在线'
  }

  return {
    displayName,
    canEdit,
    statusText,
  }
}
```

这里 `displayName`、`canEdit` 计算后不再改，用 `const` 能告诉读代码的人“这个绑定稳定”。`statusText` 会根据条件重新赋值，用 `let` 更准确。

```js
const list = []
list.push('A') // 可以修改数组内容

// list = [] // TypeError：不能重新绑定
```

这个例子说明 `const` 保护的是变量名和引用关系，不保护数组或对象内部结构。如果需要对象内容也不可变，要使用 `Object.freeze`、不可变数据结构或团队约定。

## 使用场景说明和对比

`const` 适合配置项、函数表达式、计算结果、DOM 引用、不会替换引用的数组对象。它的价值是降低读代码成本：看到 `const` 就知道这个变量名后面不会指向别的东西。

`let` 适合计数器、状态累积、条件分支后赋值、循环变量。它的价值是明确“这里会变化”，提醒读者关注变化过程。

`var` 主要出现在历史代码、老浏览器兼容代码、面试题里的作用域陷阱。新业务代码里使用 `var` 往往会削弱作用域边界。

| 场景 | 推荐声明 | 判断依据 |
| --- | --- | --- |
| 计算结果后续不变 | `const` | 绑定不需要重新赋值 |
| 分支中逐步生成结果 | `let` | 变量会在后续被改写 |
| 循环计数器 | `let` | 需要变化且应限制在块内 |
| 维护历史脚本 | 保留 `var` | 避免无关重构扩大风险 |

## 易错点提示

不要为了“未来可能会改”把所有变量都写成 `let`。如果当前代码没有重新赋值，就用 `const`，未来真需要改再调整。

不要把 `const obj = {}` 理解成“对象不能改”。对象属性可变是最常见追问。

不要说 `var` 一定污染全局。它的问题更准确地说是函数作用域、声明提升和可重复声明降低了可预测性。

## 记忆要点总结

- 默认 `const`：绑定稳定。
- 变化才 `let`：表达重新赋值。
- 少用 `var`：函数作用域和提升容易埋坑。
- `const` 管变量名，不管对象内部。

## 延伸问题

为什么很多团队 ESLint 会同时开启 `prefer-const` 和 `no-var`？

参考答案：`prefer-const` 强制把不会重新赋值的变量声明成 `const`，让变量意图更清晰；`no-var` 避免函数作用域、变量提升和重复声明带来的历史问题。两者一起用，本质上是在让声明方式承担“代码说明书”的作用。

## 可能类似的问题及简要参考答案

问：`const` 声明对象后还能改属性吗？

答：可以。`const` 禁止重新绑定变量名，不禁止修改引用值内部成员。

问：为什么循环里通常用 `let`？

答：循环计数器会变化，而且 `let` 在 `for` 循环中每轮会创建新的词法绑定，更适合闭包和异步场景。

## 辅助记忆总结

用一句话记：`const` 表示“这个名字不换人”，`let` 表示“这个值会变化”，`var` 表示“这是旧时代遗留规则”。
