# JavaScript 面试内容质量评估与优化指南

生成时间：2026-06-25

评估范围：

- `docs/interview/javascript/core`
- `docs/interview/javascript/async-modern`
- `docs/interview/javascript/es6`

参考样例：

- `docs/interview/javascript/async-modern/143-import.md`
- `docs/interview/javascript/async-modern/144-question.md`
- `docs/interview/javascript/async-modern/145-tree-shaking.md`

## 结论

现有内容已经完成了“拆分成单题 + 补齐固定章节”的第一步，但还没稳定达到复习资料应有的效果。最大问题不是缺标题，而是模板味太重、考察范围过泛、重复结构残留、示例不能独立复习。

`143-import.md`、`144-question.md`、`145-tree-shaking.md` 已在第一批优化中改成当前范例。它们现在用于证明“结论先行、机制清楚、示例可独立复习、场景可迁移”的完成态。

优化前它们的主要问题是：

- `考察范围` 写成“深度分析与补充”，无法指导复习。
- `知识点系统梳理` 内混入 `### 问题本质解读` 等重复标题。
- 文末仍有模板化记忆话术。
- 部分代码示例过长，缺少“这段代码证明什么、边界在哪里、工程里怎么取舍”的说明。

## 新评估标准

以后不再以“12 个章节是否存在”作为高分依据，而按五个复习效果判断。

| 维度 | 满分 | 判断重点 |
| --- | ---: | --- |
| 可读 | 15 | 先给直接结论，标题层级干净，段落能扫读 |
| 可用 | 20 | 有贴近前端业务的例子，能直接用于答题或代码判断 |
| 可理解 | 25 | 讲清定义、原理、执行顺序、边界条件和相近概念差异 |
| 可记忆 | 15 | 有 3-6 条高信号记忆点，而不是泛泛口号 |
| 可举一反三 | 20 | 有类似题、追问方向、场景迁移和调试线索 |
| 技术准确性 | 5 | 修正常见误区，标出版本或运行时前提 |

一票降级项：

- 只有模板话术，没有题目专属结论、边界和例子。
- `考察范围` 只写“深度分析与补充”或某个大类。
- `知识点系统梳理` 内混入重复模板标题。
- `实战应用举例` 只写“见上文示例”。
- 代码示例很长，但没有解释输入、输出、边界和取舍。

完整标准已同步到 `docs/templates/interview-answer-evaluator.md`。

## 脚本评估

### `scripts/audit-interview-content.cjs`

原脚本适合作为“章节完整度”闸门，但不够准确：

- 会把 `async-modern` 这类结构完整但仍有模板残留的文件判成 A。
- 不识别 `考察范围：深度分析与补充` 这类无效内容。
- 不识别 `知识点系统梳理` 内重复模板标题。
- 不区分单题答案和题库聚合文档。
- 报告缺少目录维度，难以安排批次。

已做最小增强：

- 支持指定审计范围：`node scripts/audit-interview-content.cjs <dir...>`。
- 跳过 `README.md` 和 `question-bank*` 聚合文件。
- 增加质量问题：考察范围过泛、知识梳理混入模板标题、模板化占位表述、实战示例不可独立复习、场景对比不可迁移、正文过长需拆例。
- 报告增加目录分布和质量问题统计。

脚本仍然不能替代人工判断技术正确性，只能用于批量发现低质模式。

### `scripts/reorganize-markdown-knowledgebase.cjs`

该脚本适合做一次性迁移、拆分和索引生成，不适合继续作为“内容质量完成”的来源。

主要问题：

- `answerDoc()` 会生成固定章节，但部分章节是模板说明，不是题目答案。
- `stripKnownSectionLabels()` 会把原始答案里的结构标题带入 `知识点系统梳理`，造成重复标题。
- `practicalSection()`、`scenarioSection()`、`memorySection()`、`similarSection()` 仍偏元说明，容易生成“看起来有内容但不能复习”的段落。
- `categoryRe` 抽到的“深度分析与补充”会被写入 `考察范围`，没有实际复习价值。

建议：不要为了这次内容优化重新跑迁移脚本。后续如果确实要重跑，应先改生成源，再迁移；否则会重新制造模板化内容。

## 目录评估

### 1. `async-modern`

现状：35 篇中 3 篇 A、26 篇 B、6 篇 C。

优点：

- Promise、async/await、模块系统等主题已有较多代码和机制说明。
- `143`、`144`、`145` 已完成范例化，可作为同主题后续优化参考。

主要缺口：

- 27 篇仍存在“考察范围过泛”。
- 32 篇仍有模板化表达和知识梳理重复标题。
- 动态 import、循环依赖、tree-shaking 这类工程题需要补构建工具、运行时失败、浏览器兼容、调试线索。

优化方式：

1. 以 `143-145` 为范例，继续处理 `141-150` 中剩余模块系统和新特性题。
2. 再回到 `116-140` Promise/async 主线，按执行顺序和错误处理重排。

### 2. `core`

现状：45 篇中 26 篇 A、1 篇 B、18 篇 C、0 篇 D。

优点：

- `020-030` 已完成范例化，闭包、this、call/apply/bind、IIFE、柯里化、回调、纯函数等函数主线题可直接复习。
- `031-045` 已完成范例化，对象创建、原型链、属性检测、属性描述符、冻结、继承、构造函数和 `new` 主线可直接复习。
- `001-019` 中有一些文件已经有较完整的业务场景和易错点。
- 类型、闭包、原型、this、new 等高频主题已经拆成单题，适合逐个优化。

主要缺口：

- `001-019` 仍有部分 C 级文件需要按同一标准补强。
- 后续 `core` 不再有 D 级文件，优先级低于 `es6` 的 70 篇 D。

优化方式：

1. 回头抽查 `001-019`，只做去模板化和小补强，避免重写已可用内容。
2. 主力转向 `es6/062-081`，继续清掉 ES6 解构、rest/spread、默认参数、模板字符串主线的 D 级文件。

### 3. `es6`

现状：70 篇中 16 篇 A、54 篇 D。

主要原因不是标题缺失，而是章节内容薄：

- `实战应用举例` 多数只是“原文已包含代码示例”。
- `使用场景说明和对比`、`易错点提示`、`记忆要点总结` 多数是占位话术。
- 箭头函数、解构、扩展运算符、class、Symbol、Map/Set 等主题缺少成体系对比。

优化方式：

1. 第一批：`046-053` 已完成，处理 `let/const/var`、块级作用域、TDZ、`const`、循环绑定、变量提升、块级函数声明。
2. 第二批：`054-061` 已完成，处理箭头函数、`this`、构造函数限制、`arguments`。
3. 下一批：`062-081`，处理解构、rest/spread、默认参数、模板字符串。
4. 第三批：`082-097`，处理 class、super、getter/setter、decorator。
5. 第四批：`098-115`，处理 Symbol、Iterator、Set/Map/WeakMap。

`es6` 不建议逐篇只补一句话。应按主题族沉淀共用对比模板，例如“语法糖 vs 运行机制”“API 能力 vs 数据结构特性”“适用场景 vs 误用边界”。

## 下一步优化指南

### 每篇的修改顺序

1. 删除或合并 `知识点系统梳理` 内重复的 `### 问题本质解读` 等标题。
2. 把 `考察范围` 改成 3-6 个具体点。
3. 在开头补一句直接结论，避免绕概念。
4. 把代码示例变成可独立复习的结构：输入、输出、边界、结论。
5. 补场景对比表：适用、不适用、替代方案、调试线索。
6. 把记忆总结压成 3-6 条具体规则。
7. 补 2-3 个类似题，答案必须能体现迁移能力。

### 每批完成标准

每批 8-12 篇，完成后运行：

```bash
node scripts/audit-interview-content.cjs docs/interview/javascript/core docs/interview/javascript/async-modern docs/interview/javascript/es6
```

合格线：

- 本批不再出现“模板化占位表述”。
- 本批不再出现“知识梳理混入模板标题”。
- `考察范围过泛` 清零。
- B 级以上文件能独立回答“结论、机制、边界、场景、类似题”。

### 优先批次

1. `es6/062-081`：处理解构、rest/spread、默认参数、模板字符串主线。
2. `async-modern/141-150`：跳过已完成的 `143-145`，其余模块系统和新特性复用同一标准。
3. `es6/062-115`：按主题族推进。
4. `core/001-019`：回头做 C 级小补强。

## 不建议做的事

- 不要为了提高脚本分数给每篇硬塞长表格。
- 不要把 `reorganize-markdown-knowledgebase.cjs` 重新跑一遍来“补齐内容”。
- 不要把题库聚合文件套进单题答案模板。
- 不要全量重写已经可用的文件；优先补缺口。
